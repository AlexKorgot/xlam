# cinematic_new

Документ фиксирует целевую архитектуру и план доработки `CinematicVideoSlider` под Figma-референс:
https://www.figma.com/design/wvU80E5h11zr2RbfUkk8yc/design?node-id=506-802&m=dev

Текущий компонент подключен через `src/components/cinematic_new/index.ts` и используется в
`src/components/ui/MainScene.tsx` как первый экран маршрутов `/` и `/main`.

## Цель

Слайдер должен выглядеть как бесконечная изогнутая кинолента:

- на экране видны ровно 3 смысловых кадра: центральный и два боковых;
- центральный кадр самый широкий и визуально главный;
- боковые кадры стоят слева и справа с зазором от центрального примерно 20-70px;
- боковые кадры обрезаются краями viewport примерно наполовину или немного больше, как в Figma;
- вся тройка лежит на общей дуге, а не выглядит как три независимые плоские карточки;
- слайды не наслаиваются друг на друга и не перекрывают видимую область соседнего кадра;
- эффект киноленты строится через общую дугу, позицию, perspective/bend и controlled gaps, а не через overlap слоев;
- переключение бесконечное, без видимого конца массива;
- переходы плавные, с ощущением прокручивания киноленты;
- слайды изогнуты, но активный кадр может распрямляться при открытии в fullscreen;
- текст и кнопки остаются DOM-оверлеем, canvas отвечает только за видео, геометрию и переход.

## Figma-ориентиры

Для десктопного кадра 1920x1080:

- фон черный/темный с мягким свечением за лентой;
- заголовок расположен над лентой, примерно `top: 100px`, размер около `60px`;
- центральный видеокадр занимает примерно 1350-1400px по ширине;
- центральная высота около 300px, формат близок к широкой киноленте;
- левый и правый кадры видимы только частично у краев экрана;
- центральный кадр находится примерно в середине экрана по Y;
- подпись активного проекта лежит поверх нижней части центрального кадра.

Точные пиксели должны пересчитываться от viewport, а не быть захардкожены только под 1920px.

## Текущая архитектура

Архитектура в целом выбрана правильно:

- `CinematicVideoSlider.client.tsx` является client component, потому что использует `window`, DOM video/canvas, события, state и GSAP.
- `SliderScene.ts` держит Three.js сцену, renderer, camera, video lifecycle, resize/visibility observers и GSAP timelines.
- `VideoPlane.ts` инкапсулирует `THREE.Mesh` с `ShaderMaterial` и uniform-состоянием одного кадра.
- `shaders/videoPlane.ts` отвечает за cover-кроп видео, изгиб, opacity, затемнение, радиус и motion distortion.
- `data.ts` содержит список слайдов.
- DOM-оверлей отделен от WebGL, что соответствует Next.js App Router и снижает риск hydration mismatch.

Эту архитектуру не нужно заменять на Embla, DOM-карусель или React Three Fiber. Для данного эффекта правильнее оставить vanilla Three.js + ShaderMaterial + GSAP timelines.

## Текущие расхождения

На момент проверки dev server на `http://localhost:3000` работает без runtime errors, canvas создается на весь viewport.
Но визуально текущий слайдер не соответствует Figma:

- сейчас центральный кадр на 1280px viewport занимает около `0.58 * width`, то есть примерно 742px; в референсе он заметно шире, около 70-72% viewport;
- шаг между кадрами считается как `centerWidth * 0.68`, из-за чего кадры перекрываются/читаются как одна растянутая видеополоса, а не как 3 отдельных кадра с зазором;
- текущее перекрытие слайдов недопустимо для целевого результата: кадры должны быть частями одной киноленты, но с видимым разделением и без наложения друг на друга;
- боковые кадры не имеют стабильного целевого правила "видно около половины или меньше";
- текущая геометрия показывает слишком много непрерывного видео по ширине и не дает чистого ощущения трех кадров киноленты;
- `opacity` дальних кадров становится `0` только после полного выхода за viewport, поэтому архитектурно может существовать больше видимых плоскостей, чем требует макет;
- при движении боковые кадры могут моргать, потому что будущий боковой кадр не имеет стабильной offscreen-buffer позиции до входа в viewport;
- `resetInactiveVideo()` сейчас может сбросить видео сразу после полного выхода кадра за viewport, поэтому при быстром обратном движении кадр может вернуться с пустым/неуспевшим обновиться `VideoTexture`;
- `opacity` меняется слишком дискретно: кадр либо видим, либо скрыт после геометрического выхода, без preload/fade зоны за краем viewport;
- desktop-адаптивность пока завязана на несколько ratio, но не имеет явного контракта "вся композиция всегда пропорциональна экрану";
- все слайды используют один и тот же video source; для эффекта "одной плёнки" это допустимо, но для проектов нужно будет либо оставить это как осознанный прием, либо заменить на реальные разные источники;
- README до этой правки описывал текущее состояние, но не задавал целевые правила Figma и критерии реализации.

## Правильное архитектурное решение

Оставить существующее разделение слоев, но изменить модель раскладки слайдов.

### 1. Ввести layout-конфиг

В `SliderScene.ts` нужен явный конфиг геометрии:

```ts
type FilmStripLayoutConfig = {
  centerWidthRatio: number;
  centerAspect: number;
  gapPx: { min: number; max: number; ratio: number };
  sideVisibleRatio: number;
  sideScale: number;
  sideRotationY: number;
  bend: number;
  edgeCurve: number;
  hiddenOffset: number;
};
```

Рекомендуемые стартовые значения для desktop:

- `centerWidthRatio: 0.70`;
- `centerAspect: 16 / 3.7` или близко к фактическому Figma-кадру;
- `gapPx: clamp(20, viewportWidth * 0.026, 70)`;
- `sideVisibleRatio: 0.42-0.5`;
- `sideScale: 0.94-0.98`;
- `sideRotationY: 0.06-0.12`;
- `bend: 38-60`;
- `edgeCurve: 16-28`.

Для mobile нужен отдельный упрощенный набор: один главный кадр и частично видимые боковые без агрессивной перспективы.

### Desktop-адаптивность

На desktop слайдер всегда должен быть пропорционален экрану. Нельзя настраивать геометрию только под один viewport.

Правило:

- все ключевые размеры считаются от `viewport.x` и `viewport.y`;
- ширина центрального кадра задается ratio от viewport, например `0.70-0.72`;
- высота считается через единый `centerAspect`, а затем ограничивается относительно высоты viewport;
- зазор между центральным и боковыми считается через `clamp(20, viewport.x * ratio, 70)`;
- боковая видимость считается либо через целевую видимую ширину, либо через целевую долю бокового кадра;
- camera Z продолжает зависеть от viewport height, как сейчас;
- при resize пересчитывается вся раскладка, без сохранения старых пиксельных значений.

Рекомендуемая модель desktop:

```ts
const centerWidth = clamp(width * 0.71, 860, width * 0.74);
const centerHeight = Math.min(centerWidth / centerAspect, height * 0.34);
const gap = clamp(width * 0.026, 20, 70);
const sideVisible = clamp(width * 0.12, 150, width * 0.18);
const sideWidth = sideVisible / sideVisibleRatio;
const sideX = centerWidth / 2 + gap + sideWidth / 2;
```

Такой расчет гарантирует:

- центр масштабируется вместе с экраном;
- зазор остается в заданном диапазоне;
- боковой кадр всегда обрезан предсказуемо;
- на широких desktop viewport композиция не расползается и не становится плоской.

### 2. Пересчитать позиции от желаемой видимости

Вместо текущего `frameStep = centerWidth * 0.68` считать боковые X-позиции так:

```ts
const gap = clamp(width * 0.026, 20, 70);
const centerWidth = width * centerWidthRatio;
const sideWidth = centerWidth * sideScale;
const sideVisible = sideWidth * sideVisibleRatio;
const sideX = width / 2 - sideVisible + sideWidth / 2;
```

Тогда:

- центральный кадр стоит в `x = 0`;
- правый кадр стоит в `x = +sideX`;
- левый кадр стоит в `x = -sideX`;
- расстояние между центральным и боковым визуально контролируется через `gap`;
- соседние кадры не заходят поверх центрального и не перекрывают его пиксели;
- боковая обрезка управляется `sideVisibleRatio`, а не случайно получается из шага.

### 3. Ограничить видимый набор тремя кадрами и добавить буфер

`getLayoutForOffset(offset)` должен явно разделять:

- `absOffset < 0.5`: центральный кадр;
- `0.5 <= absOffset <= 1.5`: боковые кадры;
- `1.5 < absOffset <= 2.5`: буферные кадры за viewport;
- `absOffset > 2.5`: полностью спящие кадры с `opacity: 0`.

Это сохранит бесконечную карусель математически, но визуально оставит только три слайда.

Буферные кадры нужны для бесшовного входа боковых слайдов. Они должны быть уже созданы, иметь валидную texture и стоять чуть за краем viewport до начала движения.
На экране при этом остаются только 3 видимых кадра: буферные кадры либо полностью за viewport, либо имеют `opacity: 0` до входа в fade-зону.

Рекомендуемая роль offset:

```ts
type SlideRole = 'center' | 'side' | 'buffer' | 'sleeping';
```

- `center`: `absOffset < 0.5`;
- `side`: `absOffset >= 0.5 && absOffset < 1.5`;
- `buffer`: `absOffset >= 1.5 && absOffset < 2.5`;
- `sleeping`: все остальное.

Для буфера лучше не добавлять новые React-слайды. Достаточно использовать существующие `VideoPlane[]` и `centeredOffset()`: при массиве из 5 слайдов роли `-2, -1, 0, +1, +2` уже доступны.
Если реальных слайдов станет меньше 5, нужно либо запретить такой набор для киноленты, либо создать loop-клоны на уровне scene model, но не DOM.

### 4. Сохранить бесконечность через `slidePosition`

Текущие `wrapIndex()` и `centeredOffset()` правильные и должны остаться.
Переходы должны анимировать непрерывное `slidePosition`, а не переставлять DOM/meshes вручную.

Во время движения `offset` становится дробным, поэтому кадры плавно переходят между ролями:

- бывший центр уходит в бок;
- боковой становится центром;
- следующий скрытый кадр входит с края.

### 5. Усилить эффект киноленты

Изгиб должен ощущаться как общая лента:

- vertex shader оставляет расчет через глобальный X (`uStripOffset + position.x * uPlaneSize.x`);
- `uBend` и `uEdgeCurve` должны зависеть от роли кадра и скорости;
- при `slideVelocity` добавлять легкий skew/crop shift, но без растягивания видео;
- верхняя и нижняя кромка должны иметь общую дугу, как в Figma.

Если зазоры между кадрами должны быть физически видимыми, их лучше получать геометрией отдельных planes, а не масками внутри одного plane.

### 6. Улучшить движение и убрать моргание

Текущее движение близко к целевому по принципу: анимируется непрерывный `slidePosition`, а layout пересчитывается на каждом update.
Улучшать нужно не сменой архитектуры, а профилем движения и lifecycle кадров.

Что изменить:

- оставить `slidePosition` как единственный источник позиции;
- добавить buffer-роль `±2`, чтобы следующий кадр входил не из `opacity: 0` в viewport, а из уже подготовленной offscreen-позиции;
- заменить жесткое `opacity: isFullyOutside ? 0 : 1` на плавную fade-зону за краем viewport;
- не вызывать `resetInactiveVideo()` сразу при первом полном выходе за viewport;
- сбрасывать inactive video только когда кадр ушел дальше buffer-зоны, например `absOffset > 2.5`, и точно не участвует в ближайшем переходе;
- держать texture буферных кадров прогретой: metadata/first frame должны быть получены заранее;
- для `slideTo()` использовать timeline с более выраженной инерцией: стартовая скорость выше, затем мягкое торможение через `power3.inOut` или кастомный ease;
- `uVelocity` не должен падать в 0 слишком рано, иначе пропадает ощущение прокрутки ленты;
- при смене active index уведомлять React ближе к прохождению центра, но texture/plane роли должны быть готовы до этого.

Практическое решение моргания:

```ts
const fadeMargin = width * 0.08;
const rightEdge = layout.x + layout.width / 2;
const leftEdge = layout.x - layout.width / 2;
const outsideRight = rightEdge - width / 2;
const outsideLeft = -width / 2 - leftEdge;
const outside = Math.max(outsideRight, outsideLeft, 0);
const opacity = 1 - smoothstep01(outside / fadeMargin);
```

При этом buffer-кадр стоит за viewport, но его mesh, texture и uniforms уже существуют. Когда он начинает входить, opacity и позиция меняются плавно.

Видео reset должен быть привязан не к `isLayoutFullyOutsideViewport(layout)`, а к роли:

```ts
if (role === 'sleeping' && wasBufferOrVisibleBefore) {
  resetInactiveVideo(index);
}
```

Это убирает мигание при быстром next/previous и при обратном движении.

### 7. Открытие оставить как morph активной плоскости

Текущий fullscreen-подход правильный:

- активный `VideoPlane` получает высокий `renderOrder`;
- его `uTransitionProgress`, `uBend`, `uCornerRadius`, `uEdgeCurve`, `uPlaneSize` анимируются в fullscreen;
- остальные плоскости уходят в opacity/depth;
- DOM details появляется после WebGL morph.

Менять это нужно только после приведения базовой киноленты к Figma.

2. Добавить desktop contract: вся геометрия считается пропорционально viewport, с `clamp()` только для min/max удобочитаемости.
3. Переписать `getLayoutForOffset()` так, чтобы центр, боковые, буферные и sleeping кадры имели явные роли.
4. Настроить desktop-геометрию под Figma: центр около 70-72% viewport, gap 20-70px, боковые видимы примерно наполовину или чуть меньше.
5. Проверить и зафиксировать отсутствие overlap: соседние слайды не должны наслаиваться на центральный или друг на друга.
6. Добавить offscreen-buffer для `offset ±2`, чтобы будущие боковые кадры уже были за viewport до начала движения.
7. Заменить жесткое исчезновение на fade-зону за viewport и не показывать больше трех кадров в видимой области.
8. Перенести `resetInactiveVideo()` с условия "полностью вышел за viewport" на условие "ушел в sleeping role за buffer".
9. Проверить, что буферные кадры имеют готовую `VideoTexture` до входа в viewport; при необходимости добавить poster/first-frame fallback.
10. Подкрутить `slideTo()`: duration/ease/`slideVelocity`, чтобы движение ощущалось как прокрутка ленты, без рывка в начале и просадки скорости перед появлением бокового.
11. Подкрутить `bend`, `edgeCurve`, `rotationY`, `darkness` и `renderOrder`, чтобы боковые читались как часть одной ленты.
12. Проверить `open()` и `close()` после новой геометрии, особенно возврат из fullscreen в точные позиции трех кадров.
13. Обновить фон/оверлей в `CinematicVideoSlider.client.tsx`: темный фон и мягкое свечение вместо плоского `#458294`, если цель строго совпасть с Figma.
14. Проверить desktop resize: 1280, 1440, 1920, wide desktop; пропорции должны сохраняться.
15. Проверить mobile отдельно: упрощенная геометрия без плотного desktop-эффекта.
16. Проверить через Playwright screenshots, canvas nonblank, console errors и Next MCP `get_errors`.
17. После кода запустить `npm run lint` и `npm run build`.

## Критерии готовности

- На desktop видны только 3 кадра.
- Центральный кадр шире боковых и визуально главный.
- Боковые кадры обрезаны краями viewport наполовину или немного больше.
- Между центральным и боковыми есть читаемый зазор 20-70px.
- Слайды не наслаиваются друг на друга; соседние кадры не перекрывают центральный.
- Переход next/previous бесконечный и плавный.
- Во время перехода появляется ощущение прокрутки киноленты, а не простого fade/slide карточек.
- Слайды изогнуты, но видео не растягивается.
- Открытие активного кадра в fullscreen не создает второй DOM-video и не ломает WebGL lifecycle.
- Нет Next.js runtime errors, browser console errors, hydration warnings.
- `npm run lint` и `npm run build` проходят.

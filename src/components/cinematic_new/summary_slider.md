# Summary: cinematic_new slider

## Задача

Нужно привести `src/components/cinematic_new` к Figma-референсу:
https://www.figma.com/design/wvU80E5h11zr2RbfUkk8yc/design?node-id=506-802&m=dev

Целевой результат: бесконечный WebGL-видеослайдер в виде изогнутой ленты/киноленты. На экране должны быть видны только 3 смысловых кадра: широкий центральный и два боковых, обрезанных краями viewport примерно наполовину или немного больше. Между центральным и боковыми нужен зазор около 20-70px. Слайды не должны наслаиваться друг на друга: соседние кадры не перекрывают центральный и не заходят поверх видимой области друг друга. Переходы должны быть плавными, с ощущением прокрутки киноленты, без моргания боковых кадров.

## Текущий статус

Текущая архитектура выбрана правильно и уже в основном реализована:

- `CinematicVideoSlider.client.tsx` держит React client boundary, DOM overlay, кнопки, keyboard/pointer handling.
- `SliderScene.ts` держит Three.js scene, camera, renderer, video lifecycle, resize/visibility observers и GSAP timelines.
- `VideoPlane.ts` инкапсулирует mesh + shader uniforms одного видеокадра.
- `shaders/videoPlane.ts` отвечает за cover-кроп, изгиб, opacity, затемнение, радиус и motion distortion.
- DOM-оверлей отделен от WebGL, что правильно для Next.js App Router и снижает риск hydration mismatch.

Архитектуру не нужно заменять на DOM carousel, Embla или React Three Fiber. Правильнее сохранить vanilla Three.js + ShaderMaterial + GSAP.

Что уже сделано в коде:

- введены роли `center | side | buffer | sleeping`;
- размеры и позиции считаются от viewport;
- `frameStep = centerWidth * 0.68` больше не используется для X-позиций;
- добавлен offscreen-buffer для `offset ±2`;
- fade/reset видео привязаны к роли, а не к первому выходу за viewport;
- `frustumCulled = false` оставлен для shader-expanded planes;
- motion использует `slidePosition`, `slideProgress`, `transitionPulse` и `slideVelocity`;
- базовый fullscreen `open()` / `close()` проверен;
- opened-навигация реализована через `openedSliding`: fullscreen plane и DOM details переключаются синхронно, без мгновенной подмены контента;
- повторный запуск details-анимации при `opening -> opened` убран;
- исчезновение details при `closing` сделано зеркальным появлению;
- после opened-навигации hidden planes нормализуются в будущие slider-layout позиции, поэтому `close()` не вытягивает боковые кадры из центра;
- UI переведен на темный фон со свечением и обновленными controls/details;
- shader сейчас работает в чистом `cover`: `containMix = 0.0`, кадрирование настраивается через `videoObjectPosition`.

## Основные проблемы сейчас

- Нужно финально проверить transition в середине движения: overlap/ощущение киноленты могут требовать донастройки на отдельных viewport.
- Нужно проверить адаптив: `1280x720`, `1440x900`, `1920x1080`, wide desktop и mobile.
- Нужно финально проверить UI overlay: heading, active label и controls не должны конфликтовать с видео.
- Нужно выполнить полный viewport sweep после всех правок: Playwright screenshots на desktop/mobile, canvas nonblank, browser console, Next MCP `get_errors`.
- Нужно заново прогнать `npm run lint` и `npm run build` именно на финальном состоянии.

## Оценка предложенных решений

### 1. Оставить текущую архитектуру Three.js + ShaderMaterial + GSAP

Вероятность успеха: высокая.

Это правильная база для эффекта киноленты. Нужны деформация плоскостей, плавный morph в fullscreen, управление видео-текстурами и синхронная анимация uniforms. DOM-карусель или Embla хуже подходят, потому что изгиб и fullscreen morph пришлось бы имитировать CSS-трансформами и масками.

Риск: средний только по производительности, если держать слишком много активных видео. Его снижает правило: играть должно только активное видео, остальные держать как paused preview/poster.

### 2. Переписать layout-модель через роли кадров

Вероятность успеха: высокая.

Роли `center`, `side`, `buffer`, `sleeping` прямо решают главную проблему: visual contract становится явным. Вместо случайного результата от `frameStep` появится управляемая композиция:

- `0`: центральный кадр;
- `-1` и `+1`: боковые видимые кадры;
- `-2` и `+2`: буферные кадры за viewport;
- остальные: sleeping.

Риск: низкий. Нужно аккуратно сохранить `centeredOffset()` и `slidePosition`, чтобы не сломать бесконечность.

### 3. Добавить offscreen-buffer для боковых слайдов

Вероятность успеха: высокая для устранения моргания.

Это наиболее важная доработка для бесшовного движения. Буферные кадры `±2` уже должны существовать, иметь готовые uniforms и texture, но находиться за viewport. Когда пользователь листает, следующий кадр входит не из пустоты, а из подготовленной позиции.

Риск: средний. Если все буферные видео будут реально играть, будет лишняя нагрузка. Правильная реализация: буферные кадры не проигрывать, но держать валидный first frame/poster/paused VideoTexture.

### 4. Перенести reset видео из viewport-exit в sleeping-role

Вероятность успеха: высокая.

Сейчас сброс при полном выходе за viewport может быть слишком ранним. Для плавной карусели кадр, который только что ушел за край, еще может понадобиться при обратном движении. Сбрасывать его нужно только после ухода за buffer-зону, например `absOffset > 2.5`.

Риск: низкий. Нужно заменить условие в `applySliderLayout()` и хранить previous role/status.

### 5. Ввести плавную fade-зону за viewport

Вероятность успеха: высокая.

Жесткое `opacity: isFullyOutside ? 0 : 1` создает риск визуального щелчка. Лучше считать opacity от расстояния за viewport через `smoothstep`. Тогда кадр мягко появляется и исчезает в зоне за краем, а не внутри видимой области.

Риск: низкий. Важно не нарушить требование "видно только 3 кадра": buffer должен оставаться за viewport или быть невидимым до входа.

### 6. Desktop-пропорциональность от viewport

Вероятность успеха: высокая.

Figma-вид можно стабилизировать через расчеты от `viewport.x` и `viewport.y`:

- центр: примерно `70-72%` ширины viewport;
- высота: через aspect и лимит от высоты viewport;
- gap: `clamp(20, viewport.x * ratio, 70)`;
- side visible: явная доля или явная видимая ширина.

Риск: средний. На wide desktop слишком широкая сцена может стать плоской, поэтому нужны проверки на 1280, 1440, 1920 и wide viewport. Возможно потребуется max-ratio или ограничение высоты.

### 7. Настроить профиль движения через GSAP

Вероятность успеха: средне-высокая.

Текущий подход с `slidePosition` уже близок к нужному. Улучшения:

- сохранить непрерывное движение `slidePosition`;
- не переставлять mesh вручную;
- скорректировать duration/ease;
- держать `uVelocity` активным достаточно долго, чтобы движение читалось как прокрутка ленты;
- уведомлять React об active slide ближе к прохождению центра.

Риск: средний. Хорошее ощущение киноленты зависит от ручной подстройки easing, bend, velocity и layout. Это нужно проверять глазами и через Playwright screenshots/video-like interaction.

### 8. Shader-изгиб оставить, но подстроить под новую раскладку

Вероятность успеха: средняя-высокая.

Текущий shader уже поддерживает глобальный X через `uStripOffset`, `uBend`, `uEdgeCurve`, `uVelocity`. Это правильная основа. Вероятно, хватит настройки uniforms. Если лента будет выглядеть как отдельные карточки, потребуется легкая правка vertex shader для более общей дуги.

Риск: средний. Слишком большой bend может визуально ломать видео, слишком маленький не даст эффекта киноленты.

## Итоговая оценка успеха

Предложенный план с высокой вероятностью приведет к целевому результату, потому что он исправляет именно первопричины:

- композиция станет заданной, а не случайной;
- desktop будет пропорционален viewport;
- боковые кадры будут обрезаться предсказуемо;
- слайды будут разделены controlled gaps и не будут наслаиваться друг на друга;
- offscreen-buffer уберет моргание;
- reset видео перестанет конфликтовать с быстрым обратным движением;
- текущая WebGL/GSAP архитектура сохранит возможность изогнутой ленты и fullscreen morph.

Основной риск не архитектурный, а визуально-настроечный: потребуется итеративно подобрать ratios, `bend`, `edgeCurve`, `rotationY`, darkness, opacity fade и ease. Это нормальный риск для такого motion/UI эффекта.

### 7. Настроить визуальный изгиб - выполнено

Результат этапа: кадры выглядят как части одной изогнутой киноленты.

Перед этапом: по запросу увеличена высота слайдов без изменения ширинных ratios. В `SliderScene.ts` изменены `centerAspect` и `maxHeightRatio` для desktop/mobile config. После последующих правок текущий desktop config использует `centerAspect: 16 / 6.4`, `maxHeightRatio: 0.54`, а tall desktop правило при viewport height от `960px` стремится к `650px`, но не выше `64%` высоты viewport и не выше ограничения от aspect.

Важная корректировка перед этапом 7: все видимые слайды должны быть выровнены по высоте. Боковые кадры больше не уменьшаются по высоте относительно центрального, потому что цельность киноленты важнее перспективного scale-down. Из этапа 3 нужно сохранить ощущение непрерывной прокрутки: кадры должны двигаться как одна последовательная лента по общей координате, но без overlap и без потери боковой видимости.

Подготовительная shader-правка перед этапом 7: добавлен `uObjectPosition` для CSS-like `object-fit: cover` / `object-position` поведения. По умолчанию используется `vec2(0.5, 0.5)`, поэтому геометрия, размеры, gap и движение не меняются. Это дает возможность позже тонко кадрировать центральный/боковые кадры без изменения layout.

Дополнительная проверка после этого этапа показала, что contain-style для активного центрального слайда помещает видео полностью, но ломает заполненность широкой ленты и дает поля. Поэтому фактическое финальное состояние shader сейчас другое: `containMix = 0.0`, все кадры идут через cover-path, а кадрирование корректируется через `videoObjectPosition`. Для движения добавлен динамический `renderOrder`, но основной контроль overlap находится в transition geometry.

## Критерии готовности

- На desktop видны только 3 кадра.
- Центральный кадр самый широкий.
- Боковые кадры обрезаны краями viewport примерно наполовину или больше.
- Gap между центром и боковыми стабилен в диапазоне 20-70px.
- Слайды не наслаиваются друг на друга.
- Движение бесконечное и плавное.
- При движении влево/вправо боковые слайды не моргают.
- Видно ощущение единой изогнутой киноленты.
- Видео не растягивается и не теряет cover-кроп.
- Fullscreen open/close не ломается.
- Opened next/previous переключает fullscreen plane и DOM content вместе.
- Закрытие после opened-навигации не вытягивает боковые кадры из центра.
- Нет hydration/runtime/browser console errors.
